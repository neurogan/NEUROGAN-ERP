# R-07 Master Manufacturing Records (MMR) Design

## Goal

Build a versioned, QA-signed MMR template library (one per finished good) that pre-populates Batch Production Records at batch initiation, providing full 21 CFR Part 111 traceability between the approved manufacturing template and every batch produced from it.

## Regulatory Basis

- **21 CFR Part 111 §111.255**: A master manufacturing record must be prepared and followed for each unique formulation and batch size.
- **21 CFR Part 111 §111.260**: The MMR must be reviewed and approved by a qualified individual before use; the approver must be a different person from the preparer.
- **21 CFR Part 11 §11.50**: Approval of the MMR (a controlled electronic record) requires a bound electronic signature with name, timestamp, and meaning = "APPROVED".
- **Audit requirement**: Each BPR must reference the exact MMR version it was produced from. FDA inspectors verify BPR execution against the MMR template.

## Architecture

### Option chosen: MMR wraps existing recipe (Option A)

`erp_mmrs` links to the existing `erp_recipes` via FK. The formula section of an MMR is the existing `erp_recipe_lines` — same rows, no duplication. Editing the formula from the MMR detail page writes to the same `erp_recipe_lines` table. The recipe view in Inventory (for FINISHED_GOOD products) is a read-through of the same data. `erp_mmr_steps` holds the process step templates. Both tables are new; no existing tables are removed.

### Layout: Split panel

MMR tab uses a left-list / right-detail split panel. Left side lists all finished goods with their MMR status. Right side shows the selected MMR detail with four sub-tabs: Formula, Process Steps, Yield, Approval.

## Data Model

### New table: `erp_mmrs`

```sql
CREATE TABLE erp_mmrs (
  id                   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id           VARCHAR     NOT NULL REFERENCES erp_products(id),
  recipe_id            VARCHAR     NOT NULL REFERENCES erp_recipes(id),
  version              INTEGER     NOT NULL DEFAULT 1,
  status               TEXT        NOT NULL DEFAULT 'DRAFT'
                                   CHECK (status IN ('DRAFT', 'APPROVED', 'SUPERSEDED')),
  yield_min_threshold  NUMERIC,
  yield_max_threshold  NUMERIC,
  notes                TEXT,
  created_by_user_id   UUID        NOT NULL REFERENCES erp_users(id),
  approved_by_user_id  UUID        REFERENCES erp_users(id),
  signature_id         UUID        REFERENCES erp_electronic_signatures(id),
  approved_at          TIMESTAMPTZ,
  created_at           TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at           TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT mmr_self_approval_prohibited
    CHECK (approved_by_user_id IS NULL OR approved_by_user_id <> created_by_user_id),
  CONSTRAINT mmr_signature_required_when_approved
    CHECK (status <> 'APPROVED' OR signature_id IS NOT NULL)
);
-- At most one APPROVED MMR per product enforced in application layer
```

### New table: `erp_mmr_steps`

```sql
CREATE TABLE erp_mmr_steps (
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  mmr_id           UUID        NOT NULL REFERENCES erp_mmrs(id) ON DELETE CASCADE,
  step_number      INTEGER     NOT NULL,
  description      TEXT        NOT NULL,
  equipment_ids    UUID[]      NOT NULL DEFAULT '{}',
  critical_params  TEXT,
  sop_reference    TEXT,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (mmr_id, step_number)
);
```

### Modified table: `erp_batch_production_records`

```sql
ALTER TABLE erp_batch_production_records
  ADD COLUMN mmr_id      UUID REFERENCES erp_mmrs(id),
  ADD COLUMN mmr_version INTEGER;
```

These columns are nullable for backwards compatibility — batches initiated before R-07 will have NULL.

## MMR Lifecycle

```
DRAFT → APPROVED → SUPERSEDED
          ↑
    (Revise creates
    new DRAFT v+1,
    old → SUPERSEDED)
```

1. **DRAFT**: Created by any user with ADMIN, QC_MANAGER, or PRODUCTION role. Formula (recipe lines), steps, and yield thresholds are editable. Cannot be used to initiate a batch.

2. **APPROVED**: Triggered by a signature ceremony. Approver must hold QC_MANAGER or QA role and must be a different user from the creator (enforced in backend, not just UI). Once approved: entire MMR is locked read-only. This version can be selected when initiating production batches.

3. **SUPERSEDED**: When an approved MMR needs revision, clicking "Revise" creates a new DRAFT at `version + 1` and marks the current APPROVED as SUPERSEDED. Historical batches retain their `mmr_id` + `mmr_version` references — they always link to the version they were produced from. Only one APPROVED MMR per product is permitted at a time.

## MMR Tab UI (Manufacturing → Master Manufacturing Records)

### Left panel — MMR library

- Lists every FINISHED_GOOD product
- Shows: product name, current version, status badge (DRAFT / APPROVED / SUPERSEDED), approver name if approved
- Products with no MMR show a "Create MMR" action in their row
- "+ New MMR" button at bottom of list

### Right panel — MMR detail

Four sub-tabs:

**Formula tab**
- Renders the associated `erp_recipe_lines` as an ingredient table (ingredient name, quantity, UOM)
- Editable in DRAFT status; read-only in APPROVED/SUPERSEDED
- "Add ingredient" / "Remove" controls shown in DRAFT only
- Footer note: "Formula locked — MMR is approved. Click Revise to make changes."

**Process Steps tab**
- Ordered list of `erp_mmr_steps`
- Each step: step number, description text, equipment required (names resolved from erp_equipment), critical parameters, SOP reference
- Editable in DRAFT (up/down buttons to reorder, add/delete steps); read-only in APPROVED/SUPERSEDED
- Equipment picker: multi-select from active `erp_equipment` records

**Yield tab**
- Min/max acceptable yield % fields
- Editable in DRAFT; read-only in APPROVED

**Approval tab**
- Shows created by, created at
- Shows approver name, signature timestamp, signature meaning if approved
- In DRAFT: "Submit for Approval" button → opens signature ceremony modal
  - Signer must be QC_MANAGER or QA role
  - Signer cannot be the creator
  - Meaning = "APPROVED"
- In APPROVED: "Revise" button → creates new DRAFT v+1, current becomes SUPERSEDED

### Header actions (APPROVED MMR)
- "View product" → navigates to the product in Inventory
- "Revise" → version increment flow

## Integration with Production Batch Initiation

Existing flow: user clicks "Start Production" → BprStartModal opens.

**Changes in R-07:**

1. **MMR lookup**: On modal open, system fetches the APPROVED MMR for the batch's product (`GET /api/mmrs?productId=:id&status=APPROVED`).

2. **If approved MMR found**:
   - Process steps pre-populated from `erp_mmr_steps` into BPR steps (copies `description`, `step_number`; `monitoring_results` pre-filled with `critical_params` as a reminder)
   - Equipment selection list pre-populated from union of `equipment_ids` across all MMR steps
   - BPR records `mmr_id` and `mmr_version` on creation
   - Info banner: "Batch will be produced from MMR v{n} — {product name}"

3. **If no approved MMR**:
   - Yellow warning banner: "No approved MMR exists for this product. Steps must be entered manually."
   - Batch can still be initiated (no hard block — avoids locking production entirely)

4. **Equipment qualification gate** (existing check, now MMR-driven):
   - For each equipment ID in the MMR step list, check `erp_calibration_schedules.next_due_at`
   - If any required equipment is overdue: hard block with specific error message listing which equipment needs recalibration
   - Cleaning currency check remains a warning (not a block) — cleaning timing is context-dependent

## API Endpoints

```
GET    /api/mmrs                          list all (with product name, status, version)
GET    /api/mmrs?productId=:id&status=:s  filter by product and/or status
POST   /api/mmrs                          create new MMR (DRAFT)
GET    /api/mmrs/:id                      get single MMR with steps
PATCH  /api/mmrs/:id                      update formula ref, yield thresholds, notes (DRAFT only)
POST   /api/mmrs/:id/approve              signature ceremony — approve MMR
POST   /api/mmrs/:id/revise               create new DRAFT v+1, current → SUPERSEDED

GET    /api/mmrs/:id/steps                list steps
POST   /api/mmrs/:id/steps                add step (DRAFT only)
PATCH  /api/mmrs/:id/steps/:stepId        update step (DRAFT only)
DELETE /api/mmrs/:id/steps/:stepId        delete step (DRAFT only)
POST   /api/mmrs/:id/steps/reorder        update step_number ordering (DRAFT only)
```

## Out of Scope (R-07)

- **Per-step equipment tracking in BPR**: equipment is tracked at batch level via `erp_production_batch_equipment_used`. Step-level equipment logging is a future enhancement.
- **MMR PDF export**: useful for paper backup / audit binder but not required for digital FDA compliance.
- **Multi-product MMRs**: one MMR per finished good only.
- **Component specification tables** (R-08): raw material spec sheets are a separate ticket.
